# CVs render via rxresu.me from a JSON Resume export; tailoring is manual

CV Variants are produced in [Reactive Resume](https://rxresu.me), not in this repo. The Achievement Bank exports the [JSON Resume](https://jsonresume.org) schema at `/resume.json` (`src/pages/resume.json.ts`); rxresu.me imports it natively. A Variant is tailored per Job Listing **in the rxresu.me editor** — dropping, reordering, and rewording Achievements — and exported to PDF there. There is no in-repo tailoring skill, no bespoke `/cv` print route, and no Playwright PDF step (all removed).

Chosen for simplicity (Michael's call). The previously-recommended path (a bespoke `/cv` print route rendered to PDF with Playwright, plus a Claude Code tailoring skill — see `docs/cv-tailoring-plan.md`) was rejected as more moving parts to build and maintain. Trade-offs: we give up the on-brand "Swiss Signal" look for CVs and the single-renderer purity of ADR-0001; we gain a battle-tested editor with templates, free PDF export, and zero rendering/tailoring code to own. ADR-0001 still holds for *content*: the Bank is the single source of truth and `/resume.json` is a view over it — no résumé facts live in two editable places.

**Consequence:** the ADR-0003 fact-lock is now a manual discipline — when editing a Variant in rxresu.me, Michael must not invent or broaden metrics, swap named tech, or add claims absent from the Bank (he is the honesty gate; nothing auto-sends anyway). The JSON Resume export is regenerated from the Bank and re-imported whenever the Bank's facts change (rxresu.me edits do **not** flow back). `tags`/`strength` on Achievements are retained as optional metadata, and `selectAchievements()` as an optional filter lever, should a pre-filtered export ever be wanted; neither is required for the manual flow.
